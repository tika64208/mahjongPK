# 自动龙岩麻将系统架构设计

## 1. 架构原则

系统采用「规则核心 + 适配层」架构：

- 规则核心只负责麻将状态、动作校验、胡牌判断、机器人决策。
- 命令行、小程序、Web、App 都只是客户端适配层。
- 后续做多端时，将规则核心放到服务端，由各端通过 API 或 socket 与服务端交互。

## 2. 第一阶段模块

```text
longyan_mj/
  tiles.py       牌定义、排序、展示、牌墙构建
  evaluator.py   胡牌判断：普通胡、七对、十三幺、金牌
  scoring.py     胡牌倍数、庄闲支付、得分结果
  rules.py       桌规配置
  state.py       玩家、碰牌、明刻等状态对象
  shanten.py     向听与有效进张评估
  bot.py         机器人出牌与碰牌策略
  game.py        牌局流程、摸牌、出牌、碰牌、胜负
  cli.py         命令行交互入口
```

## 3. 核心流程

1. 创建规则配置。
2. 创建 4 名玩家，其中 0 号为真人，1-3 号为机器人。
3. 构建并打乱 136 张牌墙。
4. 翻出 1 张金牌指示牌，本张牌从牌墙移除。
5. 发牌，庄家 14 张，其余玩家 13 张。
6. 庄家先出牌。
7. 之后玩家按座次摸牌、判断自摸、出牌。
8. 其他玩家可按座次选择碰牌。
9. 有人胡牌则结束；牌墙摸完则流局。

## 4. 胡牌算法

胡牌判断分层实现：

- 三金倒：手牌中实际金牌数量达到 3 张。
- 七对：14 张牌可组成 7 个对子，金牌可补对子。
- 十三幺：满足 13 种幺九字牌加任意对子，金牌可补缺牌或对子。
- 普通胡：按「4 组面子 + 1 对将」递归拆牌，金牌可作为万能牌补缺。
- 单游/双游 MVP：如果胡牌时仅 1 张金可作为单吊将，玩家可打出对应自然将牌进入单游；单游下轮摸牌胡，若摸到金则升级双游；双游下轮摸牌按双游胡。

对于已经碰出的玩家，普通胡牌只判断剩余暗手牌是否能组成「剩余面子 + 1 对将」。

## 5. 机器人能力分层

第一阶段提供两类机器人策略：

- `BasicBot`：基础策略，只根据对子、刻子、邻张、金牌保留价值出牌。
- `ShantenBot`：算牌策略，会分析每张候选出牌后的向听数；高手配置还会比较有效进张数量。
- `ConfigurableBot`：通过 `AbilityConfig` 组装能力，适合后续映射为可购买能力。
- `ExpertBot`：兼容包装，默认开启当前全部专家能力。

默认 3 个机器人能力不同：

- 机器人A：基础策略。
- 机器人B：向听策略。
- 机器人C：专家策略，具备剩余牌计算、胡牌价值评估、碰牌收益评估、防守风险评估和金牌策略。

机器人策略通过 `bot_policies` 注入到 `MahjongGame`，后续服务端可以按房间难度、玩家等级或付费配置切换不同 AI。

能力 ID 可作为后续权益系统的最小单位：

| 能力 ID | 说明 |
| --- | --- |
| `basic_tile_efficiency` | 基础牌效，保留对子、邻张和金牌。 |
| `shanten` | 向听计算。 |
| `effective_draws` | 有效进张种类计算。 |
| `remaining_tiles` | 结合牌河和明牌计算剩余有效张数。 |
| `hand_value` | 胡牌价值评估，偏好高分路线。 |
| `pong_ev` | 碰牌收益评估。 |
| `defense` | 防守风险评估。 |
| `gold_strategy` | 金牌保留和高价值金牌路线。 |
| `style_control` | 风格控制，可按激进、防守、高分等风格调整权重。 |
| `opponent_tenpai` | 根据牌河、明刻和剩余牌数估计对手听牌概率。 |
| `danger_refined` | 更细的危险牌评分，区分生张、熟张、字牌、中张和剩余张数。 |
| `kong_ev` | 杠牌收益评估预留，当前用于保留暗刻/杠潜力。 |
| `youjin_strategy` | 游金/双游/三游策略预留，当前用于提高金牌路线价值。 |
| `monte_carlo` | 轻量模拟收益估计，用剩余有效张和牌墙数量估算短期收益。 |
| `explanation` | 出牌解释和训练提示能力。 |

未来账号系统可保存用户购买能力，例如：

```json
{
  "user_id": "u_123",
  "bot_abilities": ["shanten", "effective_draws", "remaining_tiles", "explanation"]
}
```

游戏服务读取后调用 `AbilityConfig.from_ids(...)` 与 `build_bot_from_abilities(...)` 生成机器人。

### 5.1 专家策略能力

`ExpertBot` 当前实现了下一批核心能力：

- 剩余牌计算：通过 `BotContext.visible_counts` 统计金牌指示牌、牌河、碰牌明刻，计算有效进张还剩几张。
- 胡牌价值评估：提高金牌、七对潜力、十三幺潜力等高价值路线权重。
- 碰牌收益评估：模拟碰牌后再出牌，只有向听、剩余进张或价值不变差时才碰。
- 防守风险评估：根据对手弃牌数量、明刻数量估计压力，降低高压下的危险出牌。
- 金牌策略：金牌保留权重更高，手里有 2 张或 3 张金时显著提高打金惩罚，为三金倒和后续游金能力预留空间。
- 风格控制：通过 `BotStyle` 调整进攻、价值、防守和金牌权重。
- 对手听牌判断：结合对手弃牌数、明刻数和剩余牌墙估算压力。
- 危险牌精细评分：在粗略防守基础上额外考虑熟张、生张、字牌和中张风险。
- 杠牌收益评估：当前保留三张/四张相同牌的潜在收益，等杠流程实现后可直接升级为动作评估。
- 游金策略：当前提高金牌和对子路线价值，等游金状态机实现后可转为专门状态评分。
- 蒙特卡洛能力：当前为轻量版本，用剩余有效张 / 剩余牌墙估计短期收益；后续可替换为真实随机模拟。
- 解释能力：生成建议出牌的原因，后续可用于提示和复盘收费能力。

## 6. 计分设计

计分模块使用 `ScoreProfile` 保存底分、庄家倍数和每种胡牌的倍数表。当前 MVP 按自摸结算：

- 庄家自摸：三个闲家支付双倍。
- 闲家自摸：庄家支付双倍，其他闲家支付普通分。
- `ScoreResult` 返回四家分数变化，便于 CLI 展示，也便于后续服务端存储。

当前已实现普通自摸、七对、三金倒、十三幺、单游、双游；抢金、三游、七对游金类作为得分表预留项。

## 7. 多端演进设计

后续小程序或多端版本建议拆分为：

```text
clients/
  wechat-miniprogram/
  web/
  mobile/

server/
  api/
  websocket/
  persistence/
  bot_service/
  game_room/

core/
  rules_engine/
  evaluator/
  state_machine/
```

服务端应成为唯一可信规则执行者：

- 客户端只提交动作，例如 discard、pong、win。
- 服务端校验动作合法性并推进状态。
- 服务端广播增量事件，客户端只负责渲染。
- 每局保存 action log，便于断线恢复、复盘和纠错。

## 8. API 草案

后续服务端可提供：

- `POST /rooms` 创建房间。
- `POST /rooms/{room_id}/join` 加入房间。
- `POST /rooms/{room_id}/start` 开始牌局。
- `POST /games/{game_id}/actions` 提交动作。
- `GET /games/{game_id}` 获取当前牌局状态。
- `WS /games/{game_id}/events` 订阅牌局事件。

动作事件建议保持结构化：

```json
{
  "type": "discard",
  "player": 0,
  "tile": "M5",
  "turn": 18
}
```

## 9. 第一阶段限制

第一阶段实现的是可运行、可测试的规则引擎和命令行客户端，不处理联网、多端 UI、复杂结算、游金完整状态机。相关能力通过规则配置和状态机边界预留。
